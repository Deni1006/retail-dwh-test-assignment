## Hook к PostgreSQL

В интерфейсе Airflow (Admin → Connections) создано подключение с параметрами:

- Conn Id: postgres_dwh_raw
- Conn Type: Postgres
- Host: postgres
- Schema: dwh_raw
- Login: airflow
- Password: airflow
- Port: 5432

Для работы с PostgreSQL в Airflow установлен провайдер:
- apache-airflow-providers-postgres

В коде оператора используется PostgresHook для получения подключения:
```
from airflow.providers.postgres.hooks.postgres import PostgresHook

hook = PostgresHook(postgres_conn_id="postgres_dwh_raw")
conn = hook.get_conn()
```
Hook сам извлекает параметры подключения из созданного Connection в  Airflow UI.

## Задание 3: объяснить преимущества оператора по сравнению с функцией

### 1. Переиспользование без копипаста
Когда я писал обычную функцию, для каждого нового CSV приходилось копировать все 50 строк кода и менять в трех местах пути к файлу и название таблицы. Теперь с оператором я просто создаю новую задачу с новыми параметрами — одна строка вместо 50. Ошибок при копипасте будет меньше.

### 2. Пароли и хосты не в коде
Раньше все подключения были указаны прямо в DAG: хост, логин, пароль, БД. Если нужно было поменять пароль — лезть в код, коммитить, перезапускать. Теперь это лежит в интерфейсе Airflow в разделе Connections. Поменять можно в два клика, код не трогаем.

### 3. DAG стал читаемым
Раньше DAG был забит деталями: как читать CSV, как парсить даты, как вставлять в БД. Сейчас в DAG видно только что делает задача: читает такой-то CSV, в такую-то таблицу, через такое-то подключение. Вся сложность спрятана в операторе.

### 4. Легче тестировать
Оператор — это отдельный класс. Можно написать на него тесты, проверить DAG и оператор по отдельности. 

### 5. Единый стандарт
Если в проекте 10 разных загрузок из CSV, все они будут работать одинаково. Если нужно добавить логирование или обработку ошибок — меняю один оператор, и все задачи получают улучшение. С функциями пришлось бы править 10 разных мест.

## Задание 4. Что такое проектирование хранилища данных в методологии Anchor Modeling.

**Anchor Modeling** — это такой подход к проектированию хранилища, где всё изначально делается с расчётом на постоянные изменения и хранение истории. Смысл в том, что данные дробятся на маленькие независимые части: есть anchor (сама сущность с ID), tie (связи между сущностями), attribute (часть информации, строго типизирована) и knot (узел-список возможных состояний: пол:м/ж).

Главная идея — не трогать уже существующие таблицы. Если появляется что-то новое, мы просто добавляем новую таблицу, а не меняем старую схему. За счёт этого база спокойно проходит через изменения. А также удобно хранить историю — например, как менялись значения со временем.

Но есть и обратная сторона: таблиц становится много, запросы усложняются (много джоинов), поэтому обычно сверху делают views, чтобы с этим было проще работать. 

В итоге это подход про гибкость и адаптивную работу с изменениями. Мы можем применять его только к тем частям системы, которые быстро меняются нуждаются в использовании такого подхода.

## Задание 5. Назначение каждого из слоёв хранилища данных (RAW, ODS, MARTS).

RAW, ODS и MARTS — это слои хранилища, где данные проходят путь от сырого состояния(как пришло) до готово для аналитики и выводов.

RAW - данные попадают из источников в том виде, как есть. Ничего не чистим и не меняем.
Например: загрузили таблицу заказов из CRM со всеми дублями, кривыми датами и форматами — просто сохранили. Это нужно, чтобы всегда был оригинал и можно было переиграть загрузку.

ODS - здесь данные приводятся в нормальный вид. Убираем дубли, приводим типы, стандартизируем поля, делаем нормализацию, если необходимо.
Пример: убрали дубли заказов по order_id, привели даты к одному формату, почистили NULL — получили чистую таблицу заказов, с которой уже можно стабильно работать. 

MARTS - это витрины под конкретные задачи бизнеса или аналитики. Данные агрегируются и подгоняются под удобное использование.
Пример: сделали витрину с выручкой по дням, средним чеком по клиентам или топом товаров — чтобы BI сразу мог это использовать без сложных расчётов.

По идее можно делать витрины сразу из сырых данных. Но промежуточный слой нужен для того, чтобы нам не приходилось делать одну и ту же работу для каждой витрины(убирать дубли, пустые поля, приводить в один формат и т.д.). А для хорошего качества данных нам всегда необходимо проделывать эту работу. 
На дата-инженерах лежит большая ответственность за достоверность данных. Своими неправильными действиями мы можем сильно искажать данные или сделать ложную основу для группы аналитиков, которые по сути будут работать с тем, что мы им отдадим. Поэтому существование такий слоев позволяет обеспечить высокий уровень качества данных.

## Задание 6. Разбор запроса

### Как работает запрос сейчас:

Берётся большая таблица CM_NEW_ODN_DATA (2.7 млрд строк), джоинится с маленькой cm_code_combinations (фильтр по сегментам). Далее из поля REFERENCE25 через regexp_replace извлекается bseg_id(убираем все символы и оставляем только цифры). После этого для каждой строки выполняется несколько коррелированных подзапросов к таблице ci_bseg_SQ (~472 млн строк) для получения значений BILL_SQ и INIT_SQ с разными условиями. Логика в CASE — выбрать одно из значений в зависимости от наличия данных.

### Проблемные места:

- Много коррелированных подзапросов → таблица ci_bseg_SQ читается много раз (при такой большой таблице это дорого)
- regexp_replace вызывается несколько раз на строку → лишняя нагрузка CPU
- rtrim() в условиях → если применить функцию к полю в WHERE, индекс по этому полю НЕ будет использован.
- Нет нормального join по ключу распределения → возможен data motion между сегментами
- Повторяется одна и та же логика (одни и те же фильтры) → лишняя работа

### Как можно оптимизировать:

- Начать с самого дорогого — множественные подзапросы к большой таблице. Заменить их одним JOIN с агрегацией.
- Убрать повторяющиеся вычисления — вынести regexp_replace на уровень выше, чтобы вычислять один раз.
- Избавиться от функций в WHERE — либо очистить данные при загрузке, либо создать функциональные индексы, чтобы индексы могли работать.
- Учесть архитектуру Greenplum — проверить распределение данных, при необходимости перераспределить таблицу по ключу соединения.
- Создать индексы — по полям, которые чаще всего используются в фильтрации и JOIN.
- Измерить до и после — обязательно выполнить EXPLAIN ANALYZE до оптимизации и после, чтобы оценить эффект.
